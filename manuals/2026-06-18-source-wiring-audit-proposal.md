# Source-Wiring Audit & Refactor Proposal — 2026-06-18

Companion to `manuals/source_inventory_audit.xlsx` (generated by
`build_source_inventory.py`, static, no API calls). Triggered by the `JP_INFL1`
report: the composite was showing annual data because its `JPN_CPI` component is
served by the **World Bank annual** series, not a monthly one.

---

## 1. Root cause — why a `.csv` "source of truth" let this happen

The pipeline has **one library CSV per source** (`macro_library_fred.csv`,
`macro_library_worldbank.csv`, …). Each row maps a source series to a **served
column** (`col`, e.g. `JPN_CPI`). The served column is the *join key*, but
**nothing enforces that only one source claims a column, or that competing
claims agree on cadence / units / definition.**

Precedence is resolved **at runtime**, not declared. `_dedupe_snapshot_rows()`
groups by `(Country, Col)` and the winner is:

> has-a-value → **most-recent observation** → primary-over-aggregator (tie-break)

That heuristic is sensible *when the intended source is healthy*. It fails
silently when it isn't: for `JPN_CPI` the intended monthly FRED series
(`JPNCPIALLMINMEI`, an OECD-MEI mirror) is frozen/不鮮, so the World Bank
**annual** row — still registered as a back-fill — wins by default. The result
is a monthly column quietly serving an annual number, forward-filled for ~18
months, dragging `JP_INFL1` to an annual cadence.

Two structural gaps:

1. **No single-source-of-truth per column.** 18 served columns have ≥2 sources
   with **no `source_fallbacks.csv` entry** to declare precedence — the winner is
   whatever the runtime heuristic picks that day.
2. **No cadence/units contract.** A column's declared cadence (Monthly) is not
   checked against the cadence of the source that actually wins (Annual). And the
   FRED "replacement" for `JPN_CPI` is an *index level* (`Index 2015=100`), not a
   YoY rate — semantically wrong for the YoY blend, and nothing caught it.

## 2. Why the 2026-06-15 audit missed it

That audit verified each library row against **its own** upstream (identity /
cadence / units / slice). It never asked the *cross-row* questions:

- Do **multiple rows collide on one served column**? (JPN_CPI: WB + FRED)
- Does the **source that actually wins** match the intended cadence/definition?
- Do **composites blend components of different cadence**? (JP_INFL1)

Those are exactly what the new inventory checks.

## 3. What the inventory found (`source_inventory_audit.xlsx`)

| Scope | Finding | Count |
|---|---|---|
| Macro | Served columns with ≥2 declared sources (`FLAG_collision`) | **25** |
| Macro | …of which declared sources disagree on cadence (`FLAG_cadence_mismatch`) | **11** |
| Macro | Runtime winner ≠ declared primary (`FLAG_served_ne_primary`) | **14** |
| Macro | Multi-source columns with **no declared precedence** (`Declared_primary=AMBIGUOUS`) | **18** |
| Composites | Blends mixing component cadences (`FLAG_mixed_cadence`) | **6** |
| Market | Library rows sharing a ticker with another row (`FLAG_shared_ticker`) | **29** |

Not every flag is a bug — aggregator fallbacks *should* register alongside a
primary (that's the design). The actionable subset is the intersection:
**a coarser-cadence source winning a column that should be finer** (the JPN_CPI
class). Triage in the workbook via `Served_cadence` + `Served_last_period`:
e.g. `JPN_CPI` / `CHN_CPI` serve **Annual**; `GBR_CPI` / `CAN_CPI` collide too
but correctly serve **Monthly** (ONS / StatCan win) — healthy.

The 6 mixed-cadence composites: `JP_INFL1`, `US_INFL1` (both pull an Annual
`*_CPI` into a monthly/daily blend), `US_JOBS3`, `US_R6`, `US_INFEXP1`,
`JP_NOWCAST1` (some are legitimately mixed, e.g. weekly+monthly nowcasts — review
each).

## 4. Proposed refactor (phased)

**P0 — fix the live data bugs now.** Repoint `JPN_CPI` (and any other column
serving a coarser cadence than intended) to a monthly source. For `JPN_CPI` the
clean fix mirrors the working core component: OECD COICOP2018 **all-items YoY**
via DB.nomics (same dataflow as `JPN_CORE_CPI_YOY`). Drop the WB-annual claim on
`JPN_CPI` and the FRED *index* claim (wrong units for a YoY blend).

**P1 — make precedence explicit (single source of truth).** Every multi-source
column must have a `source_fallbacks.csv` entry declaring `t0..t3`. Target:
`Declared_primary=AMBIGUOUS` → **0**. Precedence becomes data, not a runtime
accident.

**P2 — add a CI guard** (extend `data_audit.py` or a new `validate_wiring.py`
run in the daily workflow) that **fails** on:
- a column with ≥2 declared sources but no `source_fallbacks` entry;
- declared sources for one column disagreeing on cadence/units (unless annotated
  `cadence_ok=true`);
- a **served winner whose cadence is coarser** than the best declared cadence for
  that column (the JPN_CPI smell) — this alone would have caught the bug;
- a composite whose resolved components span >1 cadence without an explicit
  `mixed_cadence_ok` annotation.

**P3 — cadence/units as a contract.** Give each served column an intended
`cadence` + `units`; the merge rejects (or down-ranks) a candidate that doesn't
match, instead of blindly taking most-recent.

**Tooling.** `build_source_inventory.py` is committed and re-runnable; regenerate
`source_inventory_audit.xlsx` on every library change (and optionally attach it
to the daily run) so the source map never drifts from reality again.

## 5. Workbook guide

- **README** — methodology + flag definitions + counts.
- **Macro indicators** — one row per served `(Country, Col)`: `Declared_primary` ·
  `Fallback_1..4` · `Actually_served` · `Served_cadence/last_period` · all
  `FLAG_*` · `All_declared` (every competing source with its `[cadence|units]`).
- **Composites** — components resolved to served cols + their cadences +
  `FLAG_mixed_cadence`.
- **Market instruments** — `index_library` ticker chains + `FLAG_shared_ticker`.
- **Collisions-\*** — pre-filtered action lists.

All sheets have frozen headers + autofilter for sort/filter review.

## 6. P1 implementation — shipped 2026-06-18

P1 (explicit precedence + runtime enforcement) is implemented for macro. The
runtime no longer decides precedence by a bare "most-recent wins" heuristic.

**Decisions taken** (with the maintainer):
- Precedence lives in a **`tier` column** added to every `macro_library_*.csv`
  (0 = national/primary/direct; 1 = aggregator — FRED/OECD/IMF/World Bank/
  DB.nomics; 2 = last-resort — Nasdaq Data Link/Alpha Vantage).
- The primary yields to the next tier when it is **missing or stale** (no obs
  within **2× its cadence**).
- **Granularity wins:** among definition-matching candidates the **finest
  cadence wins**, then lowest tier, then freshest. If the finest candidate is an
  aggregator while a coarser national source exists, that is flagged
  `PRIMARY_CADENCE_GAP` — a signal to register the finer *national* ticker
  (per the maintainer's insight). Currently: **`FRA_UNEMPLOYMENT`**.

**Selection rule** (`fetch_macro_economic._select_winner`, applied in both the
snapshot dedupe and the history merge):
1. candidates with no data are ignored; a sole candidate wins regardless of tier.
2. if candidates **mix measure-kinds** (index vs YoY vs level) it is a
   *definition collision* — the runtime does **not** guess; it falls back to the
   legacy freshest→primary pick so behaviour is unchanged until the column is
   split. Flagged `FLAG_definition_collision`.
3. otherwise: drop candidates stale by >2× their cadence vs the freshest, then
   pick finest-cadence → lowest-tier → freshest.

Covered by `test_tier_merge.py` (9 tests: cadence-first, tier tie-break,
staleness fallback, sole-candidate, definition-collision guard, period parsing).

### The one thing P1 does NOT fix on its own — the CPI columns (→ Codespace)

The 5 `_CPI` columns are **definition collisions**: a monthly **index**
(national/FRED) and the World Bank **annual YoY** share one column, so they serve
different measures across countries (`JPN_CPI`=YoY because its index feed froze;
`CHN/GBR/CAN/AUS_CPI`=index). Precedence can't fix this and the kind-guard
deliberately leaves them untouched (no regression).

**Decided fix: split by definition** — `<C>_CPI_INDEX` (national monthly index) +
`<C>_CPI_YOY` (monthly YoY), repoint the inflation composites (`JP_INFL1`,
`US_INFL1`, …) to `_CPI_YOY`, retire the World Bank annual CPI. This needs a
monthly-YoY source per country (**OECD COICOP2018 all-items YoY** via DB.nomics,
mirroring the working `*_CORE_CPI_YOY`) and a **keyed regen to validate** — both
only available in the Codespace. **This is what finally fixes `JP_INFL1`.**

**Validation note:** the merge change alters what wins for ~14 columns; confirming
the served output requires a full macro regen with API keys, which runs in the
Codespace/CI, not the web sandbox. The logic is unit-tested here; end-to-end
validation is the first step after merge.
